## Detailed Findings

### F-01 — `src/app/api/whatsapp/proposal-drafts/[id]/route.ts:41`
- **Root cause**: the codebase now has two implementations for the same API path: a direct route in `src/app/api/whatsapp/proposal-drafts/[id]/route.ts` and a registered catch-all handler in `src/app/api/_handlers/whatsapp/proposal-drafts/[id]/route.ts`. Next.js will resolve the direct route, so the reviewed catch-all protections are silently bypassed.
- **Exploit scenario**: the live direct route only checks that the caller belongs to the organization. It does not enforce the stricter auth model in the catch-all handler, it does not inherit dispatcher rate limiting, and its `POST` method does not validate CSRF at all. Any authenticated org member can refresh draft state through a service-role-backed helper by ID.
- **Fix**: remove the direct route and implement the required methods in the catch-all `_handlers` module so the path is controlled by one source of truth. If non-admin access is actually intended, add an explicit helper for that role model, plus route-level CSRF/rate limiting in the catch-all implementation.
- **Test**: add route resolution/integration tests proving `/api/whatsapp/proposal-drafts/:id` is served by the catch-all implementation, reject non-admins if admin-only, reject missing CSRF on mutating calls, and assert rate-limit headers appear.

### F-02 — `src/app/api/_handlers/integrations/places/route.ts:154`
- **Root cause**: the route authenticates the caller with `auth.getUser()` and then uses `createAdminClient()` to write org-wide integration state. It never checks whether the caller is an admin.
- **Exploit scenario**: any authenticated org member can enable Google Places, overwrite the org’s Place ID, and update connection records that drive reputation sync behavior.
- **Fix**: switch the route to `requireAdmin(request, { requireOrganization: true })`, use `auth.organizationId` instead of a service-role profile lookup, and reuse `auth.adminClient` for the write path.
- **Test**: add an integration test where a non-admin member receives `403`, while an admin can update the Place ID and the change is scoped to the correct `organization_id`.

### F-03 — `src/app/api/_handlers/integrations/tripadvisor/route.ts:11`
- **Root cause**: the route follows the same pattern as Google Places: membership-level auth plus service-role writes to `organization_settings`.
- **Exploit scenario**: any authenticated org member can replace the TripAdvisor location binding for the entire org, affecting imported reviews and downstream reputation reporting.
- **Fix**: gate both GET/POST with `requireAdmin(request, { requireOrganization: true })` and derive org context from the auth result instead of an admin-client profile lookup.
- **Test**: verify non-admin access is rejected, admin access succeeds, and the updated location binding is written only for the caller’s org.

### F-04 — `src/app/api/_handlers/admin/cost/overview/route.ts:32`
- **Root cause**: the route accumulates dozens of data-loading, aggregation, alerting, and response-formatting responsibilities inside one oversized handler.
- **Failure mode**: any change to cost metrics or alerting logic has a large regression surface, review quality drops, and tests cannot isolate failures cleanly because the core logic is not factored.
- **Fix**: extract the major query/aggregation steps into focused helpers with typed return values, keep the route handler under roughly 100-150 lines, and test each helper independently.
- **Test**: add unit tests for each extracted aggregator and an integration test that only verifies final response wiring.

### F-05 — `src/features/admin/billing/BillingView.tsx:46`
- **Root cause**: billing data fetching, derived state, plan switching, receipts, and presentation all live in one component.
- **Failure mode**: billing regressions become harder to isolate, render performance becomes opaque, and simple product changes require editing a 700+ line component with shared local state.
- **Fix**: split the component into a data hook plus dedicated sections for current plan, invoices/history, and mutation controls.
- **Test**: add focused component tests for each extracted section and hook-level tests for billing state transitions.

### F-06 — `src/app/clients/page.tsx:209`
- **Root cause**: data fetching, filtering, selection state, modal orchestration, and large table rendering all remain fused in one page component.
- **Failure mode**: changes to one client workflow can accidentally regress unrelated table/filter behavior, and memoization/performance fixes become harder to reason about.
- **Fix**: extract filter/query state into hooks and move list, detail, and modal workflows into smaller components with explicit props.
- **Test**: add unit tests for filtering hooks and component tests for table actions/modals.

### F-07 — `src/components/whatsapp/UnifiedInbox.tsx:58`
- **Root cause**: the inbox component owns conversation fetches, derived thread state, send actions, handoff/proposal flows, and multiple modal/controller concerns.
- **Failure mode**: messaging changes have a broad regression surface, stale closure/state bugs are harder to isolate, and the component remains difficult to test or optimize.
- **Fix**: extract inbox data/controller hooks and split the UI into independent conversation list, thread pane, context pane, and modal/action components.
- **Test**: add hook tests for selection/action state and component tests for each extracted pane.

## Patterns (Cross-Cutting Issues Found in 3+ Files)
- **Direct API routes bypassing dispatcher protections**: `src/app/api/availability/route.ts`, `src/app/api/whatsapp/chatbot-sessions/[id]/route.ts`, and `src/app/api/whatsapp/proposal-drafts/[id]/route.ts` do not run through `createCatchAllHandlers(...)`, so they do not inherit the generic dispatcher rate limit or JSON parsing behavior.
