## Detailed Findings

### F-01 — Stored XSS in Marketing Blog Rendering
- **Root cause**: `getPostBySlug()` loads raw `content` from `blog_posts`, `renderMarkdownContent()` transforms it with string replacements instead of a sanitizer, and the result is injected with `dangerouslySetInnerHTML`.
- **Exploit scenario**: An attacker who can author or edit a blog post stores HTML such as an inline event handler or script gadget in `content`. Any public visitor who loads `/blog/[slug]` executes that payload in the browser.
- **Fix**: Replace `renderMarkdownContent()` with a server-safe markdown pipeline. At minimum, escape raw HTML before formatting; preferably render MDX/markdown with an allowlist sanitizer and pass structured content into the component instead of prebuilt HTML strings.
- **Test**: Seed a blog post containing `<img src=x onerror=alert(1)>` or a similar inert test payload. Verify the rendered page displays escaped content and the DOM contains no executable handler/script nodes.

### F-02 — Monolithic Admin Dashboard Page
- **Root cause**: `AdminDashboard` owns session retrieval, six parallel API calls, health state, metric derivation, and the full visual tree in one client component.
- **Failure mode**: Changes to one widget or fetch path can regress unrelated dashboard behavior, stale hook dependencies become harder to spot, and focused tests are difficult to write.
- **Fix**: Extract `useAdminDashboardData()` for all async orchestration and move the UI into section components such as summary metrics, revenue panel, health panel, and recent activity. Keep the page shell responsible only for layout composition.
- **Test**: Add component tests for each extracted section and a hook test covering the loading/error/success states of the aggregated dashboard data loader.

### F-03 — Monolithic Onboarding Wizard
- **Root cause**: `OnboardingPageContent` couples multi-step form state, initial data hydration, polling, share-link state, and all step rendering in one long component, with an inline `exhaustive-deps` suppression around startup effects.
- **Failure mode**: Step-specific edits can break unrelated wizard flows, polling logic is harder to reason about, and the effect suppression can hide stale-closure regressions.
- **Fix**: Split the wizard into step components and extract hooks such as `useOnboardingData()` and `useFirstValueProgress()`. Keep polling isolated inside the progress hook so the page no longer needs a blanket hook-deps suppression.
- **Test**: Add tests that mount each step in isolation, verify state transitions between steps, and confirm the first-value polling hook starts/stops cleanly when the final step is entered or exited.

### F-04 — Oversized PDF Template Renderer
- **Root cause**: `ItineraryTemplatePages.tsx` centralizes multiple PDF template concerns in one 916-line module, and `SafariStoryPages` alone contains layout, pagination, image selection, and repeated rendering logic.
- **Failure mode**: Small layout changes can create pagination regressions or broken PDF output that are difficult to isolate because the renderers and helpers are tightly coupled.
- **Fix**: Split each template (`SafariStoryPages`, urban variant, shared footer/helpers) into dedicated modules, keep shared style constants in a common utility, and reduce each renderer to composable sections.
- **Test**: Add snapshot or golden-file PDF render tests for representative itineraries of different lengths to verify page count, section ordering, and footer placement after refactoring.

### F-05 — Monolithic Admin Insights Page
- **Root cause**: `AdminInsightsPage` owns token retrieval, eight endpoint fetch definitions, `Promise.allSettled` aggregation, toast/error logic, and all dashboard presentation in one component.
- **Failure mode**: One new insight stream or parsing branch increases complexity across the whole page, making failures harder to localize and increasing the chance of inconsistent loading/error states.
- **Fix**: Extract a typed insights loader hook and move each insight surface into a separate section/card component. Centralize JSON/error parsing so every stream follows the same contract.
- **Test**: Unit test the data hook with mixed fulfilled/rejected streams, then add component tests ensuring partial failures render degraded states without collapsing the whole page.

### F-06 — Monolithic Admin Settings Page
- **Root cause**: `SettingsPage` combines organization profile editing, WhatsApp connection management, integration status fetches, UPI state, workflow rules, and multiple side effects in a single 711-line component.
- **Failure mode**: Integration-specific changes can regress unrelated settings panels, effect lifecycles become difficult to manage, and the page is expensive to reason about or test.
- **Fix**: Split the page into focused sections such as profile settings, billing/settings, WhatsApp, and external integrations. Back each section with a dedicated hook that owns only its own fetch/mutation lifecycle.
- **Test**: Add section-level tests that mock only the relevant API surface for each settings area and verify independent loading/error handling for each extracted hook.
