## Detailed Findings

### F-01 — `src/app/api/_handlers/proposals/bulk/route.ts:12`
- **Root cause**: the route authenticates the caller with `createClient().auth.getUser()` and then upgrades to `createAdminClient()` for both the profile lookup and the proposal updates. That bypasses RLS and skips the shared `requireAdmin(...)` guard used by other proposal mutations.
- **Exploit scenario**: any authenticated org member can bulk-approve or bulk-archive proposals for the entire workspace, even if the product intends those lifecycle changes to be admin-only.
- **Fix**: replace the membership-auth flow with `const auth = await requireAdmin(request, { requireOrganization: true })`, stop creating a new admin client, and use `auth.organizationId` plus `auth.adminClient` for all reads/writes.
- **Test**: add an integration test where a non-admin member receives `403` on bulk approve/archive, while an admin can update only proposals in their own `organization_id`.

### F-02 — `src/app/api/_handlers/subscriptions/route.ts:102`
- **Root cause**: `GET /api/subscriptions` still uses the old membership-auth pattern even though billing configuration now has an admin-gated route at `src/app/api/_handlers/billing/subscription/route.ts`. The getter calls `paymentService.getCurrentSubscription(...)`, which uses `SUBSCRIPTION_SELECT` and returns the full subscription row.
- **Exploit scenario**: any authenticated org member can pull billing metadata and payment identifiers that should normally stay inside the billing/admin surface, including Razorpay IDs and free-form subscription metadata.
- **Fix**: gate `GET /api/subscriptions` with `requireAdmin(request, { requireOrganization: true })`. If non-admins truly need plan visibility, return a minimized DTO that excludes `metadata`, payment method identifiers, and Razorpay identifiers.
- **Test**: add route tests proving non-admin members get `403`, admins get `200`, and the response body only contains the intended fields.

### F-03 — `src/app/api/_handlers/admin/cost/overview/route.ts:32`
- **Root cause**: query fan-out, cache control, alert evaluation, and payload assembly are all packed into a single 760-line handler.
- **Failure mode**: any change to cost calculations or alerts carries a broad regression surface, and review quality drops because business logic is not isolated into testable units.
- **Fix**: extract cache loaders, aggregation helpers, and alert builders into typed modules under the same feature, leaving the route handler responsible only for auth, orchestration, and response formatting.
- **Test**: add focused unit tests for each extracted aggregator/alert helper and keep only a thin route integration test for end-to-end wiring.

### F-04 — `src/features/admin/billing/BillingView.tsx:46`
- **Root cause**: billing fetches, plan selection, mutations, invoices, receipts, and rendering all share one oversized local-state component.
- **Failure mode**: billing regressions become hard to isolate, render churn is harder to reason about, and small product changes require editing a 700+ line component with overlapping concerns.
- **Fix**: move billing data and mutation orchestration into a custom hook, then split UI sections into dedicated plan, usage, invoice, and receipt components with narrow props.
- **Test**: add hook tests for billing state transitions and component tests for each extracted billing section.

### F-05 — `src/app/clients/page.tsx:209`
- **Root cause**: the page still fuses data loading, filtering, selection, modal state, and large render blocks into one 703-line function.
- **Failure mode**: client-list regressions become difficult to localize, and simple workflow changes risk breaking unrelated filters or modal flows.
- **Fix**: extract filter/query state into hooks and move the list, detail drawer, and modal actions into smaller components.
- **Test**: add hook tests for filter/search behavior and focused component tests for list actions and modal flows.

### F-06 — `src/components/whatsapp/UnifiedInbox.tsx:58`
- **Root cause**: conversation loading, derived thread state, send flows, action menus, and modal coordination remain concentrated in one controller component.
- **Failure mode**: inbox changes carry a wide regression surface, state bugs are harder to isolate, and targeted performance tuning is blocked by broad re-render scopes.
- **Fix**: split the controller into data hooks plus separate conversation-list, thread-pane, and action/modal components with explicit inputs/outputs.
- **Test**: add hook-level tests for selection/action state and component tests for the extracted panes.

### F-07 — `src/app/inbox/page.tsx:56`
- **Root cause**: the broadcast workflow still lives in one 528-line function that owns audience loading, recipient filtering, schedule state, send submission, and success/error handling.
- **Failure mode**: broadcast-specific fixes can regress audience loading or schedule behavior, and test coverage stays coarse because the workflow is not decomposed.
- **Fix**: separate audience loading from send/schedule state management and move the target picker, recipient picker, and composer into focused child components.
- **Test**: add tests for audience loading, recipient selection, and send/schedule submission independently.

### F-08 — `src/app/dashboard/tasks/page.tsx:688`
- **Root cause**: the task board still keeps inline action panels, list grouping, tab state, and mutation handling in one large component path.
- **Failure mode**: task workflow changes become high-risk, and action-specific bugs are harder to cover because the state machine is spread across one large render function.
- **Fix**: extract task grouping/filter state into hooks and split each inline action flow into a dedicated component/controller.
- **Test**: add tests per inline action flow and keep the page shell focused on layout/integration only.
